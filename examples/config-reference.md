# Пример: как заполнить config под свой проект

Референс — какие значения куда подставлять. Пути, имена и числа ниже
**обобщённые**, замени на свои. Реальная конфигурация конкретного проекта
в публичный шаблон не выносится: пути к боевым базам, имена пользователей
ИБ, адреса NAS — это инфраструктурные секреты, держи их в `secrets` CI,
не в репозитории.

## Allure projects

Один Allure-проект на одну линию тестов. Типовая раскладка:

- `yaxunit-<проект>` — для unit-тестов через CI
- `bdd-<проект>` — для BDD через manual Vanessa MCP (если используешь BDD bonus)

Создаются один раз через `POST /allure-docker-service/projects` (см. шаг 1 README).

## Тестовая инфобаза

- Путь: `<путь к файловой ИБ>` — передаётся через `-BasePath` в
  `run-yaxunit-smoke.ps1`, **не хардкодится** в скрипте
- ASCII-пользователь: `ci-user` (см. Gotcha 2 — кириллица в имени ломает запуск)
- Пароль: только через `secrets.ONEC_BASE_PASS` / `$env:YAXUNIT_BASE_PASS`
- Платформа: своя версия, напр. `8.3.27.xxxx`

## Модули тестов

Передаются через `-ModuleName` / `-ModuleNames` в `run-yaxunit-smoke.ps1`.
Минимальный smoke — один модуль:

```
Тесты_СмокYAXUnit
```

Полный набор — перечисли свои модули тестов:

```
Тесты_СмокYAXUnit
Тесты_<Подсистема1>
Тесты_<Подсистема2>
...
```

## Результаты CI — что считать «зелёным»

Пример здорового прогона:

- `N/N PASSED`, ~десятки секунд end-to-end на smoke, минуты на полный набор
- Allure report: `http://<your-allure-host>:5050/allure-docker-service/projects/<project>/reports/latest/index.html`

## BDD smoke (опционально, manual через Vanessa MCP)

Если используешь `ci/vanessa-bdd-to-allure.py`:

- Несколько `.feature`-сценариев, прогон **руками** в interactive-сессии
  (не под Windows-службой — см. Gotcha 6)
- Результат заливается в отдельный Allure-проект `bdd-<проект>` тем же
  конвертером
